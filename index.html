Security Concerns in HTTP/2 Protocol

HTTP 2 is a major modification to HTTP protocol with its predecessor being standardized in 1997. HTTP 2 (standardized in 2015) was developed from SPDY protocol, Google's experimental protocol, which proved to be extremely efficient compared to HTTP 1.1. This protocol has seen tremendous acceptance rate, according to W3Techs around 11% of top 10 Million websites already supported this protocol by December 2016.

Since this is a new protocol with very high acceptance rate, we at SEFCOM Lab at ASU thought to look at the protocol from security point of view.
To properly look into this protocol we chose NGINX and Apache as web-servers and Chrome, Firefox and Safari as the client. The basic idea behind this choice was to cover most the of the web transaction and by using these we covered roughly around 80% of the traffic.

Results / Bugs (patched and notified)

We discovered a very serious Denial of Service bug in the Apache web-server. Apache HTTPD web server versions 2.4.17 through 2.4.23 did not apply limitations on request headers correctly when experimental module for the HTTP/2 protocol is used to access a resource. The net result is that a server allocates too much memory instead of denying the request. This can lead to memory exhaustion and hence Denial of Service attack on the server by a properly crafted request. Attack script for the same can be found here and all the patch related code and details can be found here.
